# AI活用記録（prompts.md）

AIとの協働の記録として、AIに依頼した内容・AIの出力・自分で確認/修正した点を残す。
「AIがこう言った」ではなく「AIはAと出したが、確認してBに直した」を書くのが目的。

## P0: 1ターミナル → web → 音

### AIへの依頼（要旨）
- テーマ: 複数AIの同時運用を音で伝える
- 最終目標: VOICEVOXで「〇〇版が承認待ち/完了です」と読み上げ ＋ Discord通知
- 今日の最小構成: 1つのターミナルから、web版を通して音を鳴らす
- 制約（軽量重視）: サーバはNode標準機能のみ（npm install不要）

### AIの出力
- 依存ゼロのNodeサーバ（`server.js`）: `/events`(SSE) と `/notify`(POST/GET) と静的配信
- 受信ダッシュボード（`public/`）: 状態ごとの音（Web Audio）＋受信ログ
- `curl` でイベントを送ると音が鳴る構成

### 自分で確認 / 修正した点

- **ページを開いても音が鳴らなかった。** 最初はAIの実装ミスを疑ったが、調べるとブラウザの自動再生制限で
  `AudioContext` が `suspended` のままなのが原因で、**ユーザー操作なしに音を鳴らせないのはブラウザ仕様**だった。
  実装バグではないので、直すのではなく **UI側に「🔊 音を有効にする」ボタンを置いて明示的に解禁する仕様**とした（初回1クリックのみ）。

- **状態ごとの音は Web Audio API で合成する方式**（音源ファイルを持たない）。
  完了=上昇チャイム（ド→ミ→ソ）／承認待ち=2連ビープ（矩形波）／エラー=下降ブザー（のこぎり波）／実行中=小さなティック音。
  実際に鳴らして聴き比べ、**画面を見ずに区別できること**を確認した。要対応（承認待ち・エラー）ほど耳に刺さる波形、
  放置してよい状態（実行中）ほど控えめ、という対応づけになっている。

- **curl で日本語のAI名を送ると 400 が返った。** `?ai=テスト版` のようにURLへ生の日本語を書いたところ、
  **サーバのコードに届く前に Node の HTTPパーサが 400 を返していた**。AIの実装バグを疑って server.js を追ったが、
  そもそも**生の日本語をURLに入れられないのが HTTP の仕様どおり**の挙動で、`--data-urlencode`（またはPOSTボディ）で
  送るのが正しい運用だと判明した。コードは直さず、代わりに**サーバが 400 を返す際に案内文を出す**よう修正し
  （「日本語は --data-urlencode かパーセントエンコードで送ってください」）、README にも同じ注意を明記した。
  `curl -G --data-urlencode ai=テスト版` なら期待どおり鳴ることを確認。

- Claude Code の `Stop` hook → `curl .../notify?state=done` を実行させる
- `Notification` hook → `state=waiting`（承認待ち）

### 実施済み（2026-07-01）: 複数AIダッシュボード

**設計判断**：サーバを「現在状態の持ち主」に変更した。
それまでのサーバはイベントを素通しするだけだったので、ブラウザを後から開くと何も表示されなかった。
サーバが各AIの最新状態を `Map` で保持し、SSE接続時にスナップショットを送るようにした結果、
「あとから画面を開いても現在の盤面が見える」ダッシュボードとして成立した。

- 同じ `ai` 名は上書き（重複して増えない）
- 要対応（承認待ち→エラー）を上に、同順位なら新しい順にソート
- 経過時間表示（1秒ごとに更新）、`/clear` で一覧リセット

**動作確認**：4AIを別状態で登録 → 新規タブで開いてスナップショットが届くこと、
既存AIの状態変更が重複せず並び替わること、新規AIが要対応で最上段に割り込むこと、
クリアで盤面のみ消えてログが残ることを、ブラウザ実機で確認。

### 実施済み（2026-07-14）: セッション追跡・実行中表示・音トグル・永続化

実運用レビューで挙がった問題への対応。

**設計判断**
- 同一プロジェクトの複数セッションが盤面上で1つに潰れる問題 → AI名を「プロジェクト名#セッションID先頭6文字」に変更（hookがstdinのJSONから `session_id` を取得）
- セッション終了の残骸が盤面に溜まる問題 → SessionEnd hook が `state=ended` を送り、サーバが盤面から自動削除
- 「今どのAIが動いているか」が見えない問題 → UserPromptSubmit hook で `state=working` を送信
- 完了音が毎ターン鳴る通知疲れ → 状態ごとの音ON/OFFトグル（localStorageに保存。試聴はOFFでも鳴る）
- サーバ再起動で盤面消失 → `.state.json` へ永続化し起動時に復元
- エラーの自動検出（PostToolUseFailure hook）は見送り。ツール失敗はClaudeが自力でリカバリする通常動作の一部で、毎回⛔＋ブザーでは誤報だらけになるため

**動作確認**
- pipe-test: 偽session_idで4 hook全部の発火・盤面反映・ended削除を確認
- 実機: 同一ディレクトリで claude -p を2本並行実行 → `ai-sound-monitor#802488` と `#925f1e` が別行で🔄実行中→✅完了→終了時に削除、のライフサイクルを確認
- サーバ再起動 → 盤面2件が復元されることを確認

**サブエージェント運用の学び**
- 通常ファイル(css/html/gitignore)の編集はサブエージェントで問題なし
- `.claude/settings.json`（hook設定）への書き込みはサブエージェントには許可されず、2エージェント連続で停止 → hook設定だけはメインセッションで実施

## P2（2026-07-14 実施）: Discord通知

### AIへの依頼（要旨）
- 完了/承認待ち/エラーを Discord Incoming Webhook に投稿。working/ended は投稿しない
- Webhook URL は環境変数 `DISCORD_WEBHOOK_URL` 優先、無ければ gitignore 済みの `discord.json` から読む（コミットに秘密を入れない）
- 5秒タイムアウト・投げっぱなし・失敗してもサーバは落とさない（VOICEVOXと同じ流儀）

### AIの出力
- `notifyDiscord()`＋`STATE_TEXT`/`STATE_EMOJI`。読み上げと同じ「番号は複数稼働時のみ」ルールで文面を生成
- `.gitignore` に discord.json 追記、README に設定手順、起動ログに 有効/未設定 の表示

### 確認・検証した点
- モックWebhookをローカルに立てて実POST内容を確認:
  - 「✅ **サウンドモニター**が完了です」（単独時・番号なし）
  - 「✅ **サウンドモニターの2番**が完了です」（2枚稼働時・番号あり）
  - 「⛔ **test**がエラーです — ビルド失敗」（message連結）
  - working は投稿されない／未設定起動では「Discord : 未設定」表示で正常動作
- ✅ 実Discordチャンネルへの投稿を確認（後述「実施済み（2026-07-15）」参照）

### workingを投稿しない理由
ブラウザ盤面は「上書きされる現在の状態」だが、Discordは「流れ去る履歴」。対話のたびに投稿すると通知として機能しなくなる。

### 実施済み（2026-07-14）: hookのテンプレート化（二重発火の予防）

**問題**：当初は `.claude/settings.json` をリポジトリに**同梱**し、「このプロジェクトで開けば鳴る」状態にしていた。
しかし実際に監視したいのは**自分が開いている全プロジェクトのAI**であってこのリポジトリではないので、
hook をグローバル（`~/.claude/settings.json`）へ移すことにした。

**AIの出力 → 直した点**：AIの初期案は同梱 hook をそのまま残してグローバルにも追加する形だった。
だが Claude Code は**グローバルとプロジェクトの hook を両方実行する**仕様なので、これではこのリポジトリで作業したときだけ
**同じ通知が2回飛ぶ（二重発火）**。「音で気づく」ツールが同じ音を2回鳴らすのは通知の信頼性そのものを損なうため、
実際に鳴らして踏み抜く前に構造として潰すことにし、**同梱 hook を削除して `claude-hooks.example.json`（テンプレート）化**、
**利用者がグローバルかプロジェクトのどちらか一方にマージする**方式に直させた（＝設定の在り処が常に1箇所に定まる）。

**実際の運用**：グローバル側に4イベント（UserPromptSubmit / Stop / Notification / SessionEnd）の hook を入れ、
全プロジェクトを監視対象にした。既存の cmux 通知 hook とは**同じイベントの `hooks` 配列に追記**して共存させている。

**動作確認**：設定は**その後に新しく開いたセッション**から有効になることを確認（既存セッションでは読まれない）。
別プロジェクト（zikken）で開いた Claude Code のセッションが盤面に載ることを確認（`names.json` に zikken が自動登録されたのが、
グローバル hook が初めて発火した記録になっている）。なお二重発火は**起こる前に構造で潰した**ので、実際に2回鳴った記録はない。

### 実施済み（2026-07-15）: 音の設定を config.json 化

**AIの出力 → 却下して直した点**：「話者・音量・速さを変えたい」と依頼したところ、
AIが最初に提案したのは**ブラウザのダッシュボードに設定UIを追加する**案だった。これは却下した。
このツールの売りは**ブラウザを開いていなくても読み上げとDiscordが動く**ことなのに、
音量を1つ変えるために localhost のタブを開かせるのでは本末転倒になる。
**JSONの数値を書き換えるだけで完結する**方式に変更させた。

**設計判断**
- `config.json`（コミット対象・既定値入り）に `voice.speaker` / `voice.volume` / `voice.speed` / `chime.volume` を持たせた。秘密は入らないので gitignore しない
- 反映はサーバ再起動。チャイム音量はブラウザが `/config` から取るので、ブラウザ再読み込みも要る
- ブラウザへ渡すのは**チャイム音量だけ**（読み上げはサーバ側で完結するため、話者・速さを渡す必要がない）
- 環境変数 `VOICEVOX_SPEAKER` / `VOICEVOX_URL` は `config.json` より優先
- 読み込みは try/catch で、**JSONが壊れていても既定値で起動する**（設定ミスでツール本体が動かなくなるのを避ける。VOICEVOX/Discordと同じ「任意機能は不在時に無害」の作法）

**動作確認**
- `config.json` の話者ID（14=冥鳴ひまり）で実際に読み上げられること。音量・速さは VOICEVOX の
  `volumeScale` / `speedScale` に、話者IDは `/audio_query` の `speaker` に渡っていることをコードで確認
- `chime.volume: 0` にすると `/config` が `{"chime":{"volume":0}}` を返す（ブラウザ側の倍率が0＝消音になる）
- `config.json` を壊れたJSONにしてもサーバは**既定値で起動する**（起動し、`/config` は既定の 1.0 を返した）

### サブエージェント運用の学び（追記）

前掲（2026-07-14）の「hook設定ファイルへの書き込みはサブエージェント不可 → メインセッションで実施」に加えて、
今回の作業で分かったこと:

- **teammate型のサブエージェントは、作業を完了しても最終報告を返さず idle になる個体がある。**
  報告を待ち続けると止まって見えるため、**成果物（ファイルの中身）を直接確認する**運用に切り替えた。
  「エージェントが報告したか」ではなく「ファイルがどうなっているか」を完了判定の基準にする。
