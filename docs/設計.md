# AI Sound Monitor 設計ドキュメント

複数のAIを並行運用するときの状態通知ツールの設計ドキュメント。
実装の写経ではなく、**なぜその形になっているか**を残すことを目的とする。

対象ブランチ: `main`（Discord Bot によるテキスト通知・ボイスチャンネル読み上げは任意機能として実装済み〈要 `npm install`〉。通知先はスラッシュコマンド（`/notify here`・`/join`）で指定する。詳細は §9 参照）

---

## 1. 概要

複数のAI（Claude Code など）を複数ターミナルで同時に走らせると、
「どのAIが終わったのか」「どれが承認待ちで止まっているのか」を人間が画面を巡回して探すことになる。
巡回コストが高いほど切り替えが遅れ、AIを遊ばせる時間が増える。

本ツールは、ターミナル側で起きた状態変化をローカルサーバに集約し、
**画面を見張らなくても分かるチャネル**（チャイム音 / 音声読み上げ / Discord通知）で人間に届ける。
ブラウザのダッシュボードは「音で気づいた後に、どれが呼んでいるかを確認する」ための補助であり、常時凝視する前提には置かない。

一言でいえば **「AIの状態を、視覚以外の感覚に逃がす」** ためのツールである。

## 2. 全体構成図

```
  ターミナル側                    ローカルサーバ (:4123)                受け取る側
 ─────────────                ─────────────────────────           ──────────────────

 Claude Code                  ┌──────────────────────────┐
  ├ UserPromptSubmit hook     │ server.js (依存ゼロ)      │
  ├ Stop hook            ──┐  │                          │
  ├ Notification hook      ├─▶│ GET/POST /notify         │
  └ SessionEnd hook        │  │   └ handleEvent()        │
                           │  │       ├ agents(Map)更新  │
 手動 curl / 他ツール    ──┘  │       ├ .state.json 保存 │        ┌────────────────┐
   curl .../notify?...        │       ├ broadcast ───────┼──SSE──▶│ ブラウザ         │
                              │       ├ speak()          │/events │  public/        │
                              │       └ sendText()       │        │ ・盤面表示       │
                              │                          │        │ ・Web Audioで鳴動│
                              │ GET /events (SSE)        │        │ ・状態別 音ON/OFF│
                              │ GET /clear               │        └────────────────┘
                              │ GET / → public/ 静的配信 │
                              └───┬──────────────┬───────┘
                                  │              │
                        ①音声合成 │              │ ②discord-bot.js（Bot・要 npm install）
                                  ▼              ▼
                    ┌──────────────────┐   ┌────────────────────────┐
                    │ VOICEVOX ENGINE  │   │ Discord Bot            │
                    │  (:50021)        │   │ ・sendText → テキスト   │
                    │ audio_query →    │   │    /notify here で指定  │
                    │ synthesis → WAV  │   │ ・playInVoice → ボイス  │
                    └────────┬─────────┘   │    /join で指定         │
                             │ 一時WAV      │ Tokenは環境変数優先      │
              ┌──────────────┴──────┐      │ → discord.json          │
              ▼                     ▼      └────────────────────────┘
   afplay で再生（macOS）    playInVoice でボイスへ流す（同じWAVを使い回す）
     → 両方の再生後に一時ファイルを削除
```

ポイントは、**ターミナル → サーバ → 各出力チャネル**の一方向フローであること。
ブラウザはサーバの状態を映すだけで、盤面の真実（source of truth）はサーバが持つ。

## 3. コンポーネント一覧

### `server.js` — 唯一のサーバプロセス（Node標準 `http` のみ、依存ゼロ）

| # | 役割 |
|---|---|
| ① | `public/` の静的配信（`/` → `index.html`）。`PUBLIC_DIR` 外へ出るパスは 403 で拒否 |
| ② | `/notify` でイベント受信（curl / Claude Code hook の入口） |
| ③ | `/events` で SSE 中継（ブラウザへ push） |
| ④ | 各AIの**現在状態**を `agents: Map` で保持し、SSE接続時にスナップショットを送る |
| ⑤ | 状態を `.state.json` に保存し、サーバ再起動後に盤面を復元 |
| ⑥ | VOICEVOX で読み上げ（直列キュー・ENGINE未起動なら自動スキップ） |
| ⑦ | `names.json` で表示名に変換し、同名が複数稼働中のときだけ番号を振る |
| ⑧ | Discord Bot への通知（テキストは `sendText`・ボイスは `playInVoice`。通知先は `/notify here`・`/join` で指定。実体は `discord-bot.js`・詳細は §9） |
| ⑨ | `config.json` から話者・音量・速さ・チャイム音量を読む（`/config` でチャイム音量をブラウザへ渡す） |

### `public/` — ダッシュボード（HTML / CSS / JS、ビルド不要）

- `index.html` … 監視盤面 / 状態と音の凡例 / 受信ログの3パネル
- `script.js` … SSE購読、Web Audio API による音の**合成**（音源ファイルを持たない）、盤面描画、状態別の音ON/OFFトグル（`localStorage` に保存）
- `style.css` … 表示スタイル

ブラウザの自動再生制限があるため、初回のみ「🔊 音を有効にする」のクリックが必要（`AudioContext` の解禁）。

### `claude-hooks.example.json` — Claude Code hook 4種の**テンプレート**（コミット対象）

| hook | 送る state | 意味 |
|---|---|---|
| `UserPromptSubmit` | `working` | 指示を出した瞬間＝AIが動き出した |
| `Stop` | `done` | 応答が完了した |
| `Notification` | `waiting` | 承認・入力待ちで止まった |
| `SessionEnd` | `ended` | セッション終了（盤面から削除） |

4つとも同型のワンライナーで、stdin の JSON から `jq` で `session_id` を取り、先頭6文字を切り出して
`ai=<プロジェクトディレクトリ名>#<session_id先頭6文字>` を組み立て、`curl -G --data-urlencode` で送る。
`-m 1`（1秒タイムアウト）＋バックグラウンド実行＋エラー握り潰しにしてあり、
**サーバが落ちていても Claude Code 側の動作をブロックしない**（監視ツールが本業を止めては本末転倒なため）。

このリポジトリは hook を**有効な設定として同梱していない**。`claude-hooks.example.json` はあくまで
コピー元のテンプレートで、利用者が次の**どちらか一方**にマージする。

| 入れ先 | 監視範囲 |
|---|---|
| `~/.claude/settings.json`（グローバル） | 全プロジェクト。複数AI同時運用の本来の使い方 |
| `<監視したいプロジェクト>/.claude/settings.json` | そのプロジェクトのみ |

**両方に入れてはいけない。** Claude Code はグローバルとプロジェクトの hook を**両方実行する**ため、
同じイベントで通知が2回飛ぶ（＝同じ音が二重に鳴る）。テンプレート方式にした理由は §7 参照。
既存の hook がある場合は、同じイベントの `hooks` 配列に**追記**すれば共存できる（cmux の通知 hook 等と併用可）。
設定は**その後に新しく開いたセッション**から有効になる。

### `config.json` — 音の設定（コミット対象）

音を変えたい人が**JSONの数値を書き換えるだけ**で済むようにした設定ファイル。秘密は入らないのでコミットする（既定値入り）。

| キー | 意味 | 既定値 |
|---|---|---|
| `voice.speaker` | 読み上げの話者ID（14=冥鳴ひまり） | `14` |
| `voice.volume` | 読み上げの音量（VOICEVOX の `volumeScale`。1.0=標準） | `1.0` |
| `voice.speed` | 読み上げの速さ（VOICEVOX の `speedScale`。1.0=標準） | `1.0` |
| `chime.volume` | ブラウザのチャイム音量の倍率（0で消音） | `1.0` |

- **反映タイミング**: サーバは起動時に1回だけ読むので、**サーバ再起動**で反映される。
  `chime.volume` だけはブラウザが `/config` から取得して使うため、加えて**ブラウザの再読み込み**が要る。
- **壊れていても止まらない**: 読み込みは try/catch で、ファイルが無い・JSONが壊れている場合は
  既定値のまま起動する（設定ミスでツール本体が動かなくなるのは本末転倒なため）。
- **環境変数が優先**: `VOICEVOX_SPEAKER` / `VOICEVOX_URL` が設定されていれば `config.json` より優先される。
- ブラウザに設定UIは**作っていない**（理由は §7 参照）。ブラウザへ渡すのは `/config` 経由の `chime.volume` だけ。

### `names.json` — 名前マップ（コミット対象）

`{"ai-sound-monitor": "サウンドモニター"}` の形式で、ディレクトリ名 → 表示名/読み名を対応づける。
盤面表示と読み上げの両方に使う。未登録のプロジェクトからイベントが届くと、
サーバが `{"名前": "名前"}` のスタブを自動追記する（人間は読み名を書き換えるだけでよい）。手編集の反映はサーバ再起動時。

### `.state.json` — 盤面の永続化（gitignore 済み）

`agents` の中身をそのまま JSON 配列で保存。イベントごとに書き出し、起動時に読み込む。

### `discord.json` — Bot Token（gitignore 済み・各自が作る）

`{ "botToken": "..." }` の形式。環境変数 `DISCORD_BOT_TOKEN` が設定されていればそちらが優先され、
どちらも無ければ Discord Bot 機能は無効のまま動く（要 `npm install`）。
Bot Token はこのツールで唯一の資格情報なのでリポジトリには置かない（§7 参照）。

### `.discord-bot-state.json` — テキスト通知先の永続化（gitignore 済み）

`{ "textChannelId": "..." }` の形式。`/notify here` で指定したテキストチャンネルIDを保存し、
サーバ再起動をまたいで通知先を覚えておく（毎回 `/notify here` を打ち直さなくてよい）。
実行時の状態でありコードではないので gitignore する。

## 4. 状態モデル

| state | 意味 | チャイム音（ブラウザ） | 読み上げ | Discord通知 | 盤面での優先度 |
|---|---|---|---|---|---|
| `waiting` | 承認・入力待ち | 880Hz 矩形波の2連ビープ（注意喚起） | 「〇〇が承認待ちです」 | ⏳ 投稿する | 0（最上位） |
| `error` | エラー | 220→164.81Hz 下降ブザー | なし | ⛔ 投稿する | 1 |
| `working` | 実行中 | 440Hz の控えめな単発ティック | なし | 投稿しない | 2 |
| `done` | 完了 | ド→ミ→ソ の上昇チャイム | 「〇〇が完了です」 | ✅ 投稿する | 3 |
| `ended` | **特別state**: セッション終了 | なし（無音） | なし | 投稿しない | 盤面から**削除** |

3つのチャネル（音・声・Discord）は**通知の重さが違う**ため、鳴らす条件も変えてある。
チャイム音は全stateに割り当てて「今どうなったか」を粒度細かく伝え、
声は行動が要る2つ（`done` / `waiting`）に絞り、
Discord は「席を外していても届く」代わりに履歴として流れ去るので、`working` を除いた3つを投稿する。

設計上の意図:

- **優先度は「人間の対応が要るか」で決めている。** 承認待ちとエラーは人間が動かないと前に進まないので最上段に来る。実行中・完了は放っておいてよいので下。
- **音の派手さも優先度に揃えている。** 待たせている状態ほど耳に刺さる音（矩形波・下降ブザー）、放置してよい状態ほど控えめな音（単発ティック）にする。
- **読み上げは `done` / `waiting` だけ。** 声は情報量が多い代わりに拘束時間が長い。「行動を促す2つ」に絞らないと、喋りっぱなしで邪魔になる。
- **`ended` は音を鳴らさない。** 終了は人間に行動を要求しないため、盤面の掃除だけ行う。
- **`error` を自動で送る hook は無い**（§7 参照）。現状 `error` は手動 curl か外部ツールからのみ発生する。

## 5. セッションの一生

```
 claude 起動
   │
   │ ユーザーが指示を送る … UserPromptSubmit hook
   ├──────────────────────────────▶ state=working  🔄 盤面に自動登録
   │
   │ 承認・入力待ちで止まる … Notification hook
   ├──────────────────────────────▶ state=waiting  ⏳ 最上段へ + 読み上げ
   │
   │ 応答完了 … Stop hook
   ├──────────────────────────────▶ state=done     ✅ + 読み上げ
   │
   │  （working → waiting → done を対話のたびに繰り返す。同じAI名は毎回上書きされ、行は増えない）
   │
   │ セッション終了 … SessionEnd hook
   └──────────────────────────────▶ state=ended    🚪 盤面から削除（無音）
```

**AI名の決まり方**

- 内部キー = `プロジェクト名#セッションIDの先頭6文字`（例: `ai-sound-monitor#802488`）。これが `agents` Map のキー。
- 表示名 = `names.json` による変換後の名前（例: `サウンドモニター`）。セッションIDは表示しない（盤面の名前にマウスを乗せると正確なキーが見える）。
- **同じ表示名が複数稼働しているときだけ**、`num`（1, 2, 3…）を付けて「サウンドモニター 2番」と区別する。
- `num` はセッション登録時に「空き番号の最小」を割り当て、そのセッションが存続する間は変わらない。2番が終了して1番と3番が残っている状態で新しいセッションが来れば、空いた2番を再利用する。番号が延々と増え続けないようにするため。

## 6. HTTP API

### `GET|POST /notify` — イベント受信

| パラメータ | 既定値 | 説明 |
|---|---|---|
| `ai` | `AI` | AI名（内部キー）。同じ名前は**上書き**され、行は増えない |
| `state` | `done` | `working` / `waiting` / `done` / `error` / `ended` |
| `message` | 空 | 補足メッセージ（任意） |

- GET はクエリ文字列、POST は JSON ボディ（パースに失敗した場合はクエリ文字列にフォールバック）。
  GET でも鳴らせるようにしているのは、hook のワンライナーやブラウザのアドレスバーから最小手数で叩けるようにするため。
- 応答は `{ ok: true, event: {...} }`。読み上げは投げっぱなしで、**応答を待たせない**（hook を1秒以内に返すため）。
- **日本語をURLに生で入れてはいけない。** `curl -G --data-urlencode` かパーセントエンコードを使う。
  生の日本語が来た場合、サーバは URL パースに失敗するがクラッシュせず 400 とその旨のメッセージを返す。

```bash
curl -G "http://localhost:4123/notify" --data-urlencode state=done --data-urlencode ai=テスト版
```

### `GET /events` — SSE（ブラウザ専用）

接続直後に現在の盤面を送り、以降はイベントを push する。3種類のメッセージ型がある。

| type | 中身 | ブラウザの動作 |
|---|---|---|
| `snapshot` | `{ type, agents: [...] }` | 盤面を丸ごと差し替える。**音は鳴らさない**（接続・クリア時に過去の音が一斉に鳴るのを防ぐ） |
| `event` | `{ type, ai, state, message, time, label, num }` | 盤面更新 + ログ追加 + 状態に応じた音 |
| `remove` | `{ type, ai, time }` | 該当AIを盤面から削除（`ended` 受信時）。音は鳴らさない |

`retry: 3000` を送っているため、サーバを再起動してもブラウザは3秒で自動再接続する。

### `GET /clear` — 盤面のリセット

`agents` を空にし、空の `snapshot` を全ブラウザへ配る。受信ログは消えない（履歴は残す）。

### `GET /config` — ブラウザ向けの音設定

`{ "chime": { "volume": 1.0 } }` を返す。ブラウザが読み込み時に1回取得し、チャイムの音量倍率に使う。
`config.json` のうち**ブラウザが必要とするのはチャイム音量だけ**なので、話者・速さ等は渡していない
（読み上げはサーバ側で完結するため）。取得に失敗した場合、ブラウザは倍率 1.0 で動く。

### `GET /` ほか — 静的配信

`public/` 配下のみを配信する（それ以外のパスは 403 / 404）。

## 7. 主要な設計判断と理由

**なぜローカルサーバを1個挟むのか**
ターミナル（イベントの発生元）とブラウザ（音を鳴らせる場所）は直接つながらない。
また「今どのAIがどの状態か」という盤面は、誰か1人が持たないと成立しない。
サーバを **状態の唯一の持ち主** に据えた結果、後からブラウザを開いても現在の盤面が見える（接続時にスナップショットを送る）。
初期実装ではサーバはイベントを素通しするだけで、ブラウザを後から開いても何も表示されなかった。この反省から今の形になった。

**なぜ依存ゼロ（npm install 不要）なのか**
個人ツールとして「clone して `node server.js` で即動く」ことを優先した。
必要なもの（HTTPサーバ・SSE・fetch）はすべて Node 標準にある。
SSE を選んだのも同じ理由で、WebSocket ライブラリを入れずにサーバ→ブラウザの push が実現できるため。通信は一方向で足りる。

**なぜセッションIDで区別するのか**
AI名をディレクトリ名だけにしていた頃、同じプロジェクトで2つのターミナルを開くと盤面上で1行に潰れ、
どちらが承認待ちなのか分からなくなった。hook が stdin の JSON から `session_id` を取り、
先頭6文字を付けて `プロジェクト名#802488` とすることでセッション単位に分離した。
先頭6文字なのは、衝突しない程度に短く、目視で追える長さだから（フルIDは長すぎて盤面が読めない）。

**なぜ番号は「複数稼働時だけ」読むのか**
セッションが1つしかないときに「サウンドモニターの1番が完了です」と読まれると、
番号が何の情報も運んでいないのに音声が長くなる。番号は**区別が必要なときだけ意味を持つ**ので、
同じ表示名が2つ以上あるときだけ付ける。単独なら「サウンドモニターが完了です」。

**なぜ読み上げはサーバ側（afplay）なのか**
ブラウザの Web Audio で喋らせると、ダッシュボードのタブを開いている間しか鳴らない。
しかし本ツールが効くのは、まさに**ブラウザを見ていない（開いてすらいない）とき**である。
サーバが VOICEVOX で WAV を合成し `afplay` で鳴らすことで、ブラウザ不要で音声通知が届く。
副次的に、読み上げは直列キューで処理され、複数AIが同時に完了しても声が重ならない。
VOICEVOX ENGINE が起動していなければ、ログを1行出して**黙ってスキップ**する（HTTP応答は正常のまま。任意機能が本体を壊さないため）。

**なぜエラーの自動検出（PostToolUseFailure hook）を見送ったのか**
ツールの失敗は、Claude が自力でリカバリする通常動作の一部である（存在しないファイルを読んで気づく、等）。
これを毎回 ⛔＋ブザーで通知すると、**誤報だらけになって音そのものが信用されなくなる**。
通知は「人間が動かないと進まないとき」だけ鳴るべきで、その条件を満たす自動シグナルが現状見つかっていない。
そのため `error` は残しつつ、自動で送る hook は置いていない（手動 curl や外部ツールから使う想定）。

**なぜ状態を永続化するのか**
サーバを再起動するたびに盤面が空になると、稼働中のセッションが「見えない状態」で走り続ける。
hook は次のイベントが起きるまで何も送らないため、自然回復しない。
`.state.json` に保存して起動時に復元することで、再起動をまたいで盤面が生き残る。
`.state.json` は実行時の状態でありコードではないので gitignore する。
ただし復元をそのまま信じると別の問題が出る。`SessionEnd` hook が飛ばない異常終了（ターミナルごと落とす等）では、
そのセッションが盤面に「実行中」のまま取り残され、次のイベントも来ないので**永遠に居座る**。
そこで起動時に一度だけ、`time` が24時間より前のエントリを掃除してから復元する（定期実行はしない）。
古い「実行中」を毎回引きずらないための最小限の後始末で、正常に終了したセッションは `ended` で既に消えているため巻き添えにならない。

**なぜ音の設定は「JSONの数値だけ」で、ブラウザに設定UIを作らなかったのか**
最初の案はダッシュボードに話者・音量・速さのUIを足すことだった。しかしこのツールの売りは、
**ブラウザを開いていなくても読み上げと Discord が動く**ことにある（§7「なぜ読み上げはサーバ側なのか」）。
音量を1つ変えるためにわざわざ localhost のタブを開かせるのでは、その利点を自分で潰すことになる。
そこで `config.json` の数値を書き換える方式にした。エディタでファイルを開いて数字を直し、サーバを再起動すれば終わる。
設定UIを持たないぶん、サーバは起動時に1回読むだけでよく、状態も増えない。
壊れたJSONでも既定値で起動するようにしてあるのは、**任意の設定が本体を壊さない**という VOICEVOX / Discord と同じ作法である。

**なぜ hook を同梱せず、テンプレートとして配るのか**
当初は `.claude/settings.json` をリポジトリに同梱し、「このプロジェクトで開けば鳴る」状態にしていた。
だが実運用では、監視したいのは**自分が開いている全プロジェクトのAI**であってこのリポジトリではない。
そこでグローバル（`~/.claude/settings.json`）へ hook を移すことにしたが、そのとき問題になるのが同梱 hook である。
Claude Code はグローバルとプロジェクトの hook を**両方実行する**ため、同梱 hook を残したままグローバルにも入れると、
このリポジトリで作業したときだけ**同じ通知が2回飛ぶ（二重発火）**ことになる。
「音で気づく」ツールが同じ音を2回鳴らすのは、通知の信頼性そのものを損なう。
そこで、実際に鳴らして確かめる前に構造として潰すことにし、同梱 hook を削除して `claude-hooks.example.json`（テンプレート）だけを配り、
**利用者がグローバルかプロジェクトのどちらか一方にマージする**方式にした。
リポジトリ側が勝手に有効な hook を持たなくなったぶん、設定の在り処が1箇所に定まり、二重発火が構造的に起きない。

**Bot Token をコミットしない扱い**
Discord Bot Token は、漏れれば **Bot そのものを他人が乗っ取れる**（接続・発言など Bot に与えた権限すべてを行使できる）、
このツールで**唯一かつ最も機微な資格情報**である。
公開リポジトリに入れれば即座に悪用されうるため、`discord.json`（gitignore 済み）または環境変数 `DISCORD_BOT_TOKEN` から読み、
リポジトリには置かない。コード上も Token の中身を**ログ・エラーメッセージに一切出さない**。
未設定なら Discord 機能は単に何もしない（VOICEVOX と同じく、任意機能は不在時に無害であること）。

## 8. 既知の制約・今後の課題

- **hook は利用者が自分で入れる（同梱していない）。** 現在の運用ではグローバル設定
  （`~/.claude/settings.json`）に4イベントの hook を入れ、**全プロジェクトを監視対象にしている**
  （既存の cmux 通知 hook とは同じイベントの配列に追記して共存させた）。
  監視対象を絞りたい場合はプロジェクト側の `.claude/settings.json` に入れる。
  **両方に入れると二重発火する**（§7 参照）ので、どちらか一方に限る点は利用者が守る必要があり、仕組みでは防いでいない。
- **サブエージェント（Task ツール）は盤面に出ない。** 現在の hook はメインセッションの
  UserPromptSubmit / Stop しか拾わないため、サブエージェントが長時間動いていても表示は「実行中」のまま。
  `SubagentStart` / `SubagentStop` 系の hook で拾える可能性があるが未検証。
- **通知先は1箇所ずつ・複数同時配信はしない。** テキストは `/notify here` で指定した1チャンネル、
  ボイスは `/join` で入った1チャンネルだけ。複数ギルド／複数チャンネルへ同時に流す機能は持たない（§9）。
- **コマンドの実行権限を絞る仕組みは無い。** Bot を招待したサーバーでは**誰でも** `/notify here`・`/join` 等を実行できる。
  実行者を制限する仕組みは未実装（個人利用の前提）。
- **複数人運用は考えていない。** 投稿は Bot 名で飛ぶため、同じチャンネルに複数人のAIが混じると
  **誰のAIが完了したのか区別しにくい**。現状は個人利用の前提。
- **効果そのものの検証が未了。** 「画面監視より早く気づけるか（状態変化に気づくまでの時間）」
  「音だけで状態を当てられるか（正答率）」は、ツールの存在意義に直結するが、まだ測っていない。
- **通知疲れ。** 対話のたびに完了音が鳴ると、音が背景化して意味を失う。
  現状は状態ごとの音ON/OFFトグル（`localStorage` 保存）で、うるさい状態だけ人間が黙らせる方式で対処している。
  「一定時間以上かかったタスクだけ鳴らす」等の自動抑制は未実装。
- **macOS 前提。** 読み上げの再生に macOS 標準の `afplay` を使っている。
  他OSでは再生コマンドの差し替えが必要（チャイム音はブラウザ側なのでOSに依存しない）。
- **VOICEVOX ENGINE の手動起動が必要。** 未起動でも動作するが、その場合は読み上げなし。
- **認証なし・localhost 前提。** `/notify` は誰でも叩ける。ローカル利用のみを想定しており、
  外部公開する場合はトークン等が必要。
- **`names.json` の手編集はサーバ再起動で反映**（起動時に1回だけ読むため）。
- **`state` の値はサーバ側で検証していない。** `/notify` は受け取った `state` をホワイトリスト照合せず
  そのまま受理・保存・配信するため、`working` / `waiting` / `done` / `error` / `ended` 以外の文字列を送っても通ってしまう。
  未知の値はブラウザ側（`script.js` の `stateOf()`）が `done` にフォールバックするので、盤面では「✅完了」として表示され、
  **実際とは違う状態を誤って伝える**可能性がある。ローカル利用（叩くのは自分だけ）の前提で許容している。

## 9. Discord Bot — テキスト通知＋ボイスチャンネル読み上げ（任意・要 `npm install`）

音も声も、**PCの前にいなければ届かない**（§1）。席を外している間の状態変化を拾うチャネルとして、
Discord への **テキスト投稿**（スマホの通知に出る）と、通話中のメンバー全員に聞こえる **ボイスチャンネルでの読み上げ** を足した。
どちらも1つの Bot（`discord.js` + `@discordjs/voice`）で行い、実装は `discord-bot.js` に隔離してある。

**なぜ通知先をスラッシュコマンドで指定するのか**

固定のチャンネルIDを設定ファイルに書く方式だと、宛先を変えるたびにファイルを書き換えてサーバを再起動する必要がある。
そこで、通知したいテキストチャンネルで `/notify here`、読み上げさせたいボイスチャンネルで `/join` を打つだけで
宛先が決まる方式にした。設定ファイルに書くのは Bot Token だけで済む。

**コマンドの構成**

| コマンド | 効果 |
|---|---|
| `/notify here` | 実行したテキストチャンネルを通知先にする（IDを `.discord-bot-state.json` に保存） |
| `/notify off` | テキスト通知を止める |
| `/join` | 実行者が今いるボイスチャンネルに Bot を参加させる |
| `/leave` | ボイスチャンネルから退出する |

- コマンドは**ギルド（サーバー）単位で登録**する。グローバル登録は反映に最大1時間かかるため、
  `ClientReady` 時に参加済み全ギルドへ登録し、起動後に新しいサーバーへ招待された場合（`GuildCreate`）もその場で登録する（再起動なしで使える）。
- `/join` は接続に数秒かかりうるので `deferReply({ ephemeral: true })` で「考え中」を出してから接続し、成否を `editReply()` する。
- 返信はすべて ephemeral（実行者だけに見える）にして、チャンネルに操作ログを撒かない。

**投稿の中身**

`handleEvent()` の中で、`STATE_TEXT` に載っている state（`done` / `waiting` / `error`）のときだけ `sendText()` で投稿する。
文面は読み上げと同じ「誰が」（`who`）を共有しており、同名セッションが複数稼働しているときだけ番号が付く。`message` があれば ` — ` で連結する。

```
✅ **サウンドモニター**が完了です                 ← 単独稼働（番号なし）
✅ **サウンドモニターの2番**が完了です            ← 同名が複数稼働しているときだけ番号
⛔ **test**がエラーです — ビルド失敗              ← message は「 — 」で連結
```

AI名を `**太字**` にしているのは、Discord のモバイル通知が本文を先頭から数十文字しか表示しないため。
**誰が・どうなったか**が最初に来る語順にして、通知バナーだけで判断できるようにしている。

**テキスト通知先の永続化（`.discord-bot-state.json`）**

`/notify here` で決めた通知先チャンネルIDは `.discord-bot-state.json`（gitignore 済み）に保存し、
サーバ再起動後に `initBot()` が読み戻す。毎回 `/notify here` を打ち直さずに済むようにするため。
一方、ボイス接続（どのVCにいるか）は永続化しない。再起動したら `/join` し直す運用でよく、過剰実装しない。

**なぜ動的 import なのか（依存ゼロ方針を壊さないため）**

このツールの土台は「clone して `node server.js` で即動く／`npm install` 不要」（§7）である。
Bot のためだけに `discord.js` 一式を常時 import すると、この機能を使わないユーザーにまで `npm install` を強制してしまう。
そこで `discord-bot.js` の `initBot()` は、**Bot Token が設定されているときだけ** `await import('discord.js')` する。未設定なら import すらしない。
設定はあるのに `npm install` していない場合は、動的 import が投げる例外を握りつぶし、
「`npm install` が必要です」のログを1行出して**機能だけ無効化する**（サーバ本体は止めない）。
これは VOICEVOX ENGINE 未起動時に黙ってスキップするのと同じ、「任意機能は不在時に無害」（§7）の作法である。

**Bot Token はこのツール唯一の資格情報であること**

外部に出す秘密は Bot Token 1つだけである。
Token が漏れると **Bot そのものを乗っ取れる**（接続・発言など Bot に与えた権限すべてを他人が行使できる）ため、機微さが高い。
`botToken` は **環境変数（`DISCORD_BOT_TOKEN`）が最優先、無ければ `discord.json`** から読み、リポジトリには置かない（`discord.json` は gitignore 済み。§7 参照）。
コード上も、Token の中身を**ログ・エラーメッセージに一切出さない**（ログイン失敗時も「失敗しました」までしか出さない）。

**なぜ同じ WAV を使い回すのか**

`server.js` の `speakOnce()` は、VOICEVOX で合成した WAV を一時ファイルに書き、`afplay`（ローカル再生）で鳴らしている。
ボイスチャンネル再生のために**もう一度 VOICEVOX へ合成を投げる**のは、同じ音声を2回作る無駄であり、2つの再生で声質や生成タイミングがずれる原因にもなる。
そこで `speakOnce()` は1回だけ合成した WAV のパスを、ローカル再生（`playFile`）とボイスチャンネル再生（`playInVoice`）の**両方に渡す**。
両者は `Promise.allSettled` で並行実行し、**両方の再生が終わってから**一時ファイルを削除する（片方が失敗しても、もう片方の再生と後片付けは走る）。
なお WAV をボイスチャンネルへ流すには Opus へのエンコードと 48kHz PCM への変換が要るため、`@discordjs/voice` が `opusscript`（純JS の Opus 実装）と `ffmpeg-static`（同梱の FFmpeg バイナリ）を使う。
音声の暗号化には `libsodium-wrappers`（WASM・純JS）を使う。いずれも**ネイティブビルドが要らない**ものを選んでおり、`npm install` でコンパイルに詰まらないようにしている。

**なぜ接続を維持し続けるのか（`/join` から `/leave` まで）**

読み上げのたびにボイスチャンネルへ「参加 → 発話 → 退出」を繰り返す作りにもできるが、参加・退出には毎回ネットワーク往復と待ちが発生し、
チャンネルへの入退室が通知欄でチカチカと目立つ。
本ツールは対話のたびに短い読み上げが何度も発生するため、**`/join` で1回接続したらそのまま居座り、必要なときに喋る**方が単純で速い。
`AudioPlayer` も1つだけ作って接続に購読させておき、再生はこのプレイヤーが自然に直列化する（読み上げキュー〈§7〉自体が直列なので、実際には同時再生も起きない）。

**既知の制約（実装した範囲を正直に）**

- **macOS 前提は変わらない。** ローカル再生は引き続き `afplay`。ボイスチャンネル再生は OS 非依存だが、本体が macOS 前提のまま。
- **通知先は1箇所ずつ。** テキストは `/notify here` の1チャンネル、ボイスは `/join` の1チャンネルだけ。複数ギルド／複数チャンネルへの同時配信は持たない（§8）。
- **コマンドの実行権限は絞っていない。** Bot を招待したサーバーでは誰でも `/notify here`・`/join` 等を実行できる（§8）。
- **接続失敗時のリトライは基本的な再接続のみ。** 切断（`VoiceConnectionStatus.Disconnected`）を検知したら、公式ドキュメント推奨の `entersState` で数秒だけ自然回復を待つ。戻らなければ接続を破棄し、**再参加は `/join` に委ねる**（指数バックオフ等の作り込みはしていない）。
- **実チャンネルでの疎通は未確認。** `node --check` による構文確認と、Token 無しでのサーバ起動（クラッシュせず「未設定」ログが出て `/notify` が正常応答し、`sendText()` が no-op になること）までは確認したが、実際に Bot Token を用いてログイン・スラッシュコマンド登録・テキスト投稿・ボイス接続・発話するところまでは未検証（Token が要るため）。
